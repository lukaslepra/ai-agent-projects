## 复盘问题

### 1. 流式输出和普通输出在代码上的核心区别是什么？`delta.content` 和 `message.content` 有什么不同？

核心区别是 `stream=True`，以及取回复的方式不同：
- 普通输出：`response.choices[0].message.content` — 一次性拿到完整文字
- 流式输出：用 `for chunk in response` 循环，每次从 `chunk.choices[0].delta.content` 取一小片文字

`delta.content` 是"这次新吐出来的片段"，可能是一个字、几个字，也可能是 None（最后的结束 chunk）。`message.content` 是最终完整的回复，流式模式下没有这个字段。

### 2. 为什么 Chat Completions API 本身是无状态的？多轮对话是怎么实现"记忆"的？

API 无状态是因为服务器不存储任何会话信息——每次请求对它来说都是全新的，它不知道你上次说了什么。

多轮对话的"记忆"完全靠客户端手动维护：把每轮的 user 消息和 assistant 回复都 append 进 messages 列表，下次调用时把完整列表传进去，AI 才能"看到"之前的对话。本质上不是记忆，是每次都把历史重新喂给它。

### 3. 滑动窗口策略有什么缺陷？什么场景下会出问题？你会怎么改进？

缺陷：早期的重要信息会被丢掉。比如第1轮告诉 AI "我是合同审查系统的管理员，有最高权限"，聊了10轮后这条信息被滑出去了，AI 就不记得了。

容易出问题的场景：
- 用户在开头设定了特殊角色或规则，后面引用时 AI 已经忘了
- 长对话中早期提到的关键数据（如合同条款编号）后面需要用到

改进方式：重要信息放进 system prompt 而不是对话历史，system 消息不会被滑出去。或者超出阈值时把旧历史摘要成一段话插回去，而不是直接删掉。

### 4. `print(delta, end="", flush=True)` 里的 `flush=True` 是干什么的？去掉会有什么现象？

Python 的 `print` 默认是有缓冲区的，不会每次都立刻把内容输出到屏幕，而是积攒一批再一起输出。`flush=True` 强制每次打印后立刻刷新缓冲区，字立刻出现在屏幕上。

去掉 `flush=True` 的现象：流式输出会"失灵"——不再是一个字一个字出现，而是等缓冲区满了或者程序结束才一次性全部打印出来，跟普通调用看起来没区别，打字机效果消失。

### 5. 面试题（Week 3 第2题）：你们的聊天机器人是怎么处理上下文长度限制的？

**Situation**：在做合同风险审查 Agent 时，合同本身就很长，加上多轮确认对话，messages 很快超出 token 限制。

**Task**：需要在不丢失关键信息的前提下，控制每次请求的 token 消耗。

**Action**：采用了两层策略——① 核心规则和角色定义放在 system prompt，这部分永远保留不裁剪；② 对话历史用滑动窗口只保留最近5轮，超出部分在丢弃前先用一次 API 调用生成摘要，把摘要拼回 system prompt 里；这样既控制了长度，又不会丢掉重要上下文。

**Result**：每次请求的 token 数稳定在预期范围内，早期设定的审查规则始终有效，用户在长对话后也不会遇到 AI "忘事"的情况。

## 今日关键收获

1. **流式输出核心是 `stream=True` + 迭代 chunk**，`flush=True` 是打字机效果的关键，不能省
2. **多轮对话 = 客户端手动维护 messages 列表**，API 本身无状态，记忆靠传历史实现
3. **滑动窗口的坑**：重要信息要放 system prompt，不能只放对话里
4. **练习4发现**：短问题耗时反而占比大，瓶颈在网络延迟而不是生成速度
