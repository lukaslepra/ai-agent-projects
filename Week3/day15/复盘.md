## 复盘问题

### 1. finish_reason 有哪几种值？生产环境里最需要处理哪种情况，为什么？

finish_reason 常见的值有四种：
- `stop`：正常结束，AI 把话说完了
- `length`：被 max_tokens 截断，AI 话说到一半被强制停了
- `tool_calls`：模型要调工具，回复里不是普通文字而是工具调用信息
- `content_filter`：内容被安全过滤，reply 可能是空的

生产环境里**最需要处理 `length`**。原因是：如果回复被截断，特别是 JSON 格式的回复，只有一半的 JSON 直接导致 `json.loads()` 报错，整个流程崩掉。合同风险分析器这类场景，分析结果本来就是一段 JSON，被截断后解析失败，用户什么都看不到。

### 2. temperature=0 和 temperature=1 各适合什么场景？合同风险分析器温度设成1.0会有什么问题？

- `temperature=0`：适合结构化输出、分类、代码生成、合同分析等需要确定性答案的场景。每次调用结果几乎相同。
- `temperature=1.0`：适合创意写作、头脑风暴，输出多样化，但不稳定。

合同风险分析器如果温度设成 1.0，问题很严重：
1. 同一份合同分析两次可能得到不同结论
2. JSON 格式输出可能时而正常时而乱掉
3. 法律场景要求精准一致，高温度会引入不必要的随机性，用户无法信任结果

### 3. 强 prompt 和弱 prompt 的核心差距在哪里？五段式结构里哪一段最关键？

核心差距在于**对 AI 行为的约束程度**。弱 prompt 只告诉 AI "你是谁"，但没说要怎么做、输出什么格式、遇到奇怪问题怎么处理。AI 只能靠自己猜，结果自然随意。

五段式结构里我觉得**格式（Format）那一段最关键**。因为：
- 角色定义影响的是语气和领域知识
- 但格式定义决定输出能不能被程序解析
- 在 Agent 开发里，下游代码要读 AI 的输出，格式不对就是硬报错
- 今天的实验也验证了这一点：加了 JSON 格式约束之后，强 prompt 100% 可被 `json.loads()` 解析

### 4. MonitoredClient 在实际项目里有什么价值？合同审查项目里什么情况下会用到它？

**价值**：让每次 API 调用可观测。不只是"调了一次"，而是知道花了多少 token、多少钱、哪次调用最贵。

**合同审查项目里的使用场景：**
- 一份合同可能需要多次调用（分块分析、摘要、风险评级），用 MonitoredClient 可以统计整份合同的总成本
- 上线后监控每天的 token 消耗，发现异常（某个用户上传了超长合同）时能及时告警
- 做成本优化决策时有数据依据，比如发现 completion_tokens 占大头，就可以压缩输出格式

### 5. 面试题（Week 3 第1题）：你在项目里是怎么控制 LLM 输出质量的？

**Situation**：在做合同风险审查 Agent 时，要求 AI 输出结构化的 JSON 风险报告，下游直接 `json.loads()` 解析。

**Task**：需要确保每次 AI 输出格式稳定、内容不随意发散、边界情况有兜底处理。

**Action**：采用了三个手段——① 写五段式 system prompt，明确角色、规范、格式、示例和约束；② 把 temperature 设为 0，让输出高度确定；③ 每次调用后检查 `finish_reason`，如果是 `length` 就在日志里告警并通知用户内容不完整，避免解析崩溃。

**Result**：JSON 解析成功率从约 70% 提升到接近 100%，边界情况（恶意请求、无关问题）也能正确转人工处理，不会乱回答。
